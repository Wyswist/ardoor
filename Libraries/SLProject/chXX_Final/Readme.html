/*! \mainpage SLProject Welcome to the SLProject. SL stands for Scene Library and is a 3D computer graphics scenegraph library. It is developed at the Berne University of Applied Sciences and is used for student projects. This demo application shows you what you can learn in one semester about 3D computer graphics in real time rendering and ray tracing. During the semester we built this framework in C++ and OpenGL ES 2.0 so that it can run also on mobile devices. The framework runs without changes on Windows, Linux, Mac OSX, Android (>=2.3.3) and Apple iOS (>=5). The framework can render alternatively with ray tracing which provides in addition high quality transparencies, reflections and soft shadows. Please visit the project homepage at http://code.google.com/p/slproject/ The source code is provided under the GNU General Puplic License (GPL). The code is provided without any warranties whether expressed or implied.

Framework Features:

  • Platform independent real time 3D rendering with OpenGL ES 2.0 on Windows, Mac OSX, Linux, Android and Apple iOS5.
  • 99% of the code is in platform independent C++ or GLSL (OpenGL Shading Language).
  • Hierarchical scenegraph with grouping and referencing.3DS file format import.
  • Sorted alpha blending for transparencies.
  • View frustum culling for optimal rendering of large scenes.
  • Standard GLSL shaders for vertex lighting, pixel lighting, bump mapping, parallax mapping, water and glass shading.
  • Turntable and first person camera animation.
  • Stereo rendering with anaglyphs, side-by-side, line-by-line and checkerboard image separation.
  • Simple 2D GUI buttons for hierarchical menus.
  • Space partitioning with axis aligned bounding volume hierarchy combined with uniform grids.
  • Object picking with fast ray casting.
  • Alternative rendering with ray tracing. On desktop OS the ray tracing is optimally parallelized using OpenMP.
  • Distributed ray tracing for anti aliasing, soft shadows and depth of field.

New: Minimal OpenGL ES2 App in 4 Languages

  • Minimal OpenGL application in C++, C#, Java & WebGL.
  • The projects can serve as entry points for OpenGL programming newbies.
  • The apps show a simple rectangle with texture that can be rotated with the mouse implemented in:
    • C++ with the GLFW library for GUI.
    • C# with the OpenTK GL wrapper.
    • Java with the JOGL GL wrapper.
    • WebGL in JavaScript
  • The code in the 4 Languages is as identical as possible.

Framework Structure:

  • The applications starting code depends on the operating system. In all cases we have the most outer shell of the application that handles the window and the OpenGL context creation and passes then the events to a C-function interface before it is handled by the C++-framework. The following OS are supported:
    • Windows, Linux & Max OSX applications use the GLFW library for the platform independent window and context creation.
    • The Android application starts in JAVA and passes the events with JNI (Java Native Interface) to the C-interface.
    • On Apple iOS Devices the application starts in ObjectiveC before it passes the events to the C-interface.
  • The framework uses beside OpenGL the libraries JPEG, PNG and ZLIB for image loading and nvwa for memory management.
  • The toplevel class of the framework is SLScene that represents the scene with its properties. Different scenes are created in SLScene::onLoad.
  • SLSceneView represents a dynamic real time 3D view onto the scene. If provides the major event handlers.
  • The scenegraph hierarchy is built with the classes SLNode, SLShape, SLMesh, SLGroup, SLRefShape and SLRefGroup.
  • The classes SLRevolver, SLSphere, SLCylinder, SLCone, SLBox, SLPolygon and SLRectangle are all inheritants from SLMesh.
  • The space partitioning is implemented in the classes SLAABBox and SLUniformGrid.
  • All OpenGL ES 2.0 code is restricted to the classes SLGLState, SLGLTexture, SLGLBuffer, SLGLShader and SLGLShaderProg.
  • The lighting is implemented in the classes SLMaterial, SLLight, SLLightSphere and SLLightRect.
  • The classes for the 2D-GUI are SLTexFont, SLText and SLButton.
  • The ray tracer is implemented in the classes SLRaytracer and SLRay.
  • The linear algebra math is implemented in the classes SLMat3, SLMat4, SLVec3 and SLVec4.

    • Math: All math classes for vectors and matrices
    • MeshLoader: Classes for 3DS mesh file loading.

Notes on the Android Project:

  • You need at least a device with Android 2.3.3 (API-10) with a graphic accelerator that supports OpenGL ES 2.0.
  • The project uses the VS-Android project definition that calls the Apache-ANT tool for building the JAVA layer and the apk (Android Package).
  • You need to install prior JAVA SDK and the Android SDK and NDK. After installation you can update the Android SDK with the Android SDK Manager (by starting the {AndroidSDKPath}/tools/android.bat).
  • Run the VS-Android installer (_external\vs-android\MSBuild\install.cmd) and follow the installation and usage tutorial.
  • Connect an Android device with the USB cable.
  • Check the connection with the command "adb devices" in the console. It starts the USB deamon and returns the device ID. The Android Debug Bridge (adb.exe) is located in {AndroidSDKPath}/plattform-tools. If there is no device ID listed you have a USB connection problem. For my Samsung Galaxy S2 it was not enough to install the Google USB driver from the SDK. I had also to install the Samsung Kies desktop application that installs its own USB driver.
  • The VS project is in general a static library project that is compiled with the GCC arm compiler provided in the Android NDK.
  • Start the build process by building the chXX_Final_Android project. It compiles the C/C++ code, the JAVA classes and builds, signs and installs the apk on the device.
  • Until now I was not able to use the debugger via the adb (Android Debug Bridge).
  • The project has a prebuild step that copies all the assets (textures, models and shaders) into the apk folder (_globals/GUI/Android/apk/assets).
  • When the app runs the first time it extracts the compressed asset files into the apps folder on the device.
  • The debug configuration has a postbuild step that starts the logging with adb (adb logcat SLProject:I *:S).
  • In the debug configuration the apk is automatically signed with a development certificate.
  • The realease version has to be properly signed with your certificate. Please read the Android dev docs.
  • You can find the demo project also on the Android Market
  • Final remark on Android development: I'm a newbie on Android development. After several days of struggeling and installing and reinstalling you really feel the nerdy linux background of the Android system. The qualtiy of the graphics driver is good on the block buster selling devices such as the Samsung Galaxys etc. But otherwise you get tons of GLSL compilation errors on rarer devices. There seems to be no debugging options for native apps. You just can log on the output stream and read that with adb. Native code development in Eclipse with deployment is a nightmare. The VS-Android project was a salvation.

Notes on the Apple iOS Project:

  • You need an Apple computer with at least Mac OS 10.7 installed.
  • You need at least XCode version 4 that you can download at developer.apple.com.
  • Without registering for the iOS Developer Program (99$ per year) you can only run the application on the iOS emulator.
  • With a developer account you can debug, run and install the application on any device with the iOS version 5 or newer.
  • You need a iOS device that supports OpenGL ES 2.0. These are iPod Touch 2nd generation or newer, iPhone 3 or newer and all iPads.
  • Please refer to the iOS online documentation on how to create the provisioning for the application.
  • The XCode environment provides also sofisticated tools for profiling and for the OpenGL ES pipeline analysis.
